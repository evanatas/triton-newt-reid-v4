## Что доказано (на held-out, не на dev)
1. **local-feature ≫ global-embedding.** Ядро 0.79 против MegaDescriptor 0.30 и DINOv2 0.17 на ТЕХ ЖЕ кропах и протоколе.
   McNemar ядро vs Mega **b=3/c=52**, vs DINOv2 **b=0/c=62** (ядро выигрывает ВСЕ 62 расходящиеся пробы), p<0.001.
   → провал 3.0 (0.188) обусловлен **парадигмой global-эмбеддинга**, а не только плохими кропами.
2. **Рычаги подтверждены:** affine-деформ-верификация (вместо жёсткой гомографии) + rolling multi-reference gallery.
3. **Переобучения нет:** test (0.79) ≈ dev (0.753), CI сильно перекрываются ([0.70–0.86] vs [0.66–0.83]); gap не отрицательный.
   Обобщение на dev: PW 1.0, LAB (другой домен) rolling 0.679.

## Честные оговорки (для защиты ВКР)
- **KPI подаётся раздельно (rolling vs single).** 0.790 — режим накапливаемого каталога (каждая прежняя съёмка → референс,
  прод-сценарий заказчика с ежемесячными съёмками); 0.700 — нижняя граница при единственном прежнем снимке. **0.79 ≠ one-shot.**
- **top-5 ниже амбиции.** Контрактный порог заказчика — **top-1 ≥ 75 % (взят: 0.79)**. Формализованная исполнителем планка
  **top-5 ≥ 95 %** (заказчик по кейсу — «точность > 75 %», без типа метрики) **не взята**: **R@5 = 0.92** (−3 п.п.). Подаём
  честно: главный контрактный порог взят, top-5 близок, но ниже 95 % (потолок TK temporal при узоре, растянутом между съёмками).
- **Срез within-TK.** Финал и демо считаются на целевой temporal-когорте TK (Карелина); PW (ребристый, same-session,
  лёгкий протокол) в headline-KPI и демо-каталог не входит — headline это именно TK temporal, как и требует KPI перепоимки.
- **2/3-мес интервалы = 1.0** — на малом n (Wilson-нижняя ниже); rolling-механика легитимна (probe видит более ранние съёмки той же особи).
- **open-set** — n_new=17, недомощно; трактовать как индикатор, не как финальную verification-метрику.
- **learned-матчеры** отклонены предварительно: DISK (урезанный/частичный прогон) не бьёт SIFT и ×230 медленнее; ALIKED не оценён (баг kornia).

## Метод (замороженное ядро)
BiRefNet-сегментация тела → кроп 512×512 с нормализацией масштаба + маскирование метки-номера → gray+CLAHE(2.0) →
**SIFT** (1500 keypoints, Lowe 0.75) → **affine-деформ-верификация** (`cv2.estimateAffine2D`, 6-DOF, RANSAC thr 5) →
**rolling multi-reference** gallery (max-pool по особи). Полная спецификация: `core/CORE.md`, `core/core_config.json`.

## Честность протокола (подтверждено внешним ревью + гейтами)
Сплит по `individual_id` (identity-disjoint train/dev/test); gallery = ранняя сессия, probe = поздние; sealed-манифест
(sha256, вскрыт один раз через `mark_opened`); все методы для финальной таблицы заморожены ДО вскрытия (анти-fishing);
на тесте зарепортированы ВСЕ. Гейты честности G1–G7 / G-S1…G-S7 — зелёные. Внешний аудит F-1…F-14 исполнен до вскрытия.

## Интерпретируемость (Часть 9 — что «видит» ядро)
SIFT-оверлеи повторных съёмок (dev, `core/interpret.py` → `core/artifacts/interpret/interpret_montage.jpg`): у одной и той
же особи **affine-inlier-соответствия (что считает ядро) концентрируются на узоре пятен брюшка** — медиана 20 inliers,
8 разных особей, интервалы 1–2 мес; у разных особей (cross) — лишь **3–4** разбросанных. Наглядно подтверждает: ядро
матчит индивидуальный узор пятен, а не силуэт/фон/лапы. (Зелёные линии = affine-inliers, серые = отклонённые Lowe-матчи.)

## Open-set: формализация (known / на проверку / new)
**Требование ТЗ (ФТ-4):** пометить «новая особь», если максимальная оценка сходства ниже калиброванного порога. ML-часть (порог/классификатор) — этап 4.0; занесение в БД — продукт.

**Метрики** (`core/metrics.py`): TAR = доля known принятых (score ≥ порог); FAR = доля new ложно принятых; FRR = 1−TAR; **BAKS** = TAR (сбаланс. точность на известных), **BAUS** = 1−FAR (на неизвестных), **G = √(BAKS·BAUS)** — целевая open-set-метрика (штрафует дисбаланс жёстче среднего).

**Честный порог (индекс Юдена на DEV, не на test):** порог = **9 inliers** выбран на dev (`core_config.json:open_set_DRYRUN`, зафиксирован ДО вскрытия, 2026-06-30) → dev BAKS 0.598 / G 0.773; на sealed-тесте Юден дал **тот же порог 9** → подача честна, порог не подгонялся сверх dev. Воспроизводится `core/openset.py` → `artifacts/openset_dev.json`.
*Метод. долг для будущего sealed:* `youden_threshold` в коде пересчитывается на каждом fold (docstring обещает фиксацию dev-порога); совпадение dev = test = 9 спасло — в новой версии передавать dev-порог явной константой.

**Финальный результат (sealed, заморожен):** порог 9 → **BAKS 0.67 / BAUS 1.0 / FAR 0.0 / G 0.819**, n_new = 17.

**Чистота среза (сильнее 3.0):** все «новые» на sealed — **того же вида TK (Карелина)**, что и известные → open-set 4.0 = **within-TK**, конфаунда состава по виду НЕТ. (В 3.0 «новые» были другого вида PW, sealed AUROC 0.446 был спутан составом, а within-cohort срез невычислим.)

**Tri-state (known / на проверку / new):** на dev within-TK порог 9 разделяет known/new **чисто** (FAR = 0, review-зона вырождена — метрически достаточно одного порога). В демо (`backend/reid_service.verdict`) «на проверку» задаётся в шкале **калиброванной уверенности** (не сырые inliers) как human-in-the-loop подушка: known (уверенность ≥ 70 % И отрыв ≥ 8 п.п.) / new (< 40 %) / на проверку (между). Две шкалы (inliers-метрика vs уверенность-UX) — не смешивать.

**Честные оговорки:** n_new = 17 «модест» — индикатор, не финальная verification-метрика; BAUS 1.0 — свойство выборки (новые — реальные TK-особи при малом n), не гарантия на потоке; open-set подаётся в rolling-режиме (0.79), нижняя граница single-ref — 0.70.
