# 🦎 triton 4.0 — дорожная карта

> Цель версии: **top-1 ≥ 75 % при перепоимке особи** (temporal re-id). Фокус 4.0 — **ML до метрики**.
> БД особей, карточки ИК/КВ, UI — отдельный этап ПОСЛЕ метрики, в части 4.0 не входят.

> ⚠️ Это проект по **компьютерному зрению** (re-identification по изображениям). Предметной областью
> (биология тритонов) мы не занимаемся.

## Установка версии (определяет всё)

- **Версия 3.0 считается провалившейся** (TK temporal top-1 = 0.188, KPI не взят). Смотрим в неё **только**
  чтобы понять, как делать НЕ надо.
- **Код и артефакты 3.0 НЕ переиспользуем** (ни `triton_crop`, ни `triton_data`, ни манифест/кропы).
  Строим **новое, более сильное решение** с нуля на современном стеке.
- Переиспользуем только **сырые данные заказчика** (TK / PW / LAB / GCN) и **уроки-антипаттерны**.

на то, что реальные данные содержат воспроизводимый между сессиями сигнал узора.

## Общий план — 11 частей в 5 фазах

Декомпозиция **работы** (не кода). Каждая часть реализуется **заново**. Метрику реально двигают части **5, 6, 7**
(+ данные 2–3); остальное — фундамент, честность, интерпретируемость, защита.

| # | Часть | Фаза | Двигает top-1? |
|---|---|---|---|
| 1 | Согласование scope, данных и KPI с заказчиком | 0. Рамка | нет (де-рискер) |
| 2 | Сбор и единый реестр данных (TK+PW+LAB+новые) | 1. Данные | косвенно (данные) |
| 3 | Temporal-протокол и сплиты по `individual_id` | 1. Данные | косвенно (данные) |
| 4 | Сегментация брюшка (BiRefNet / SAM 2) | 2. Препроцессинг | усилитель |
| 5 | Ориентация + нормализация деформации узора (приём Hoque) | 2. Препроцессинг | **усилитель** |
| 6 | **Диагностика сигнала во времени (спайк LAB)** | 3. Диагностика | **гейт-потолок** |
| 7 | **Time-invariant представление узора** | 4. Ядро | **ядро метрики** |
| 8 | Open-set: известный / новый / на проверку | 4. Ядро | нет (FAR≈0) |
| 9 | Матчинг пятен + интерпретируемость (оверлей, тепловая карта) | 5. Оценка | низкий |
| 10 | Бейзлайны и сравнение (AmphIdent/MiewID/Wild-ID/I3S) | 5. Оценка | нет (для ВКР) |
| 11 | Honest evaluation + финальный sealed | 5. Оценка | нет (честность) |

**Порядок исполнения:** `6 (спайк) → 1,2,3 → 4,5 → 7 → [гейт] → 8,9,10 → 11`.
Спайк (6) идёт первым как дешёвая разведка; согласование (1) — параллельно (LAB уже на руках).

Что нового против 3.0 (6 блоков): согласование данных (1), протокол отдельно (3), нормализация деформации (5),
диагностический спайк (6), open-set отдельно (8), бейзлайны/сравнение (10).

Каждая часть планируется **отдельно** перед исполнением. Детальные планы — по мере подхода к части.

## Первый шаг (по исполнению) — Часть 6: диагностический спайк на LAB

**Вопрос:** есть ли воспроизводимый между сессиями сигнал узора — и какой метод представления ловит его лучше?
Объединяет де-рискер (часть 6) с разведкой для ядра (часть 7).

**Кратко:** новый изолированный код (`triton 4.0/spike_lab/`, без импортов 3.0) → LAB temporal-бенчмарк
(gallery = ранняя сессия особи, probe = поздние, по интервалам) → сегментация брюшка (BiRefNet/SAM 2) →
прозвон представлений (MegaDescriptor / MiewID / DINOv2 + local-feature SuperPoint+LightGlue) → cross-session
identity-level recall@1/5/10 + FRR/FAR по интервалам + визуальный аудит → **вердикт-гейт** в журнал.

**Критерий готовности:** таблица метод × интервал с Wilson CI; ≥10 визуальных оверлеев перепоимок; письменный
вердикт (сигнал есть/нет, лучший метод, реалистичный потолок, рекомендация для части 7).

Детальные планы разрабатывались отдельно перед каждой частью.

## Антипаттерны 3.0 (не повторять)

random-сплит (инфляция +36 п.п.); fold по кадру вместо особи (identity-disjoint 47/65); разный препроцессинг
gallery/probe (урок H1, ±0.20); поворот базы (ломает эмбеддер); зеркалирование; вывод «метод не работает» без
sensitivity-проверки; «точность» одной цифрой вместо набора метрик retrieval + verification; дообучение на
имитации дрейфа (McNemar b=c=0); «+0.21 от дообучения» (артефакт препроцессинга).
